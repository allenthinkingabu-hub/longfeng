## 一、本次升级的核心诉求（来自 2026-04-23 讨论）

v1.7 的格局是「事前 + 事后」：

- **事前**：sc-phase-mapping 确认 SC ↔ Phase 映射关系
- **事后**：business-match 报告 + `--aggregate` 全项目汇总

**v1.7 的盲区**：**事中执行过程**不受约束，AI 可以"写一坨通用代码然后回填锚点糊过门禁"；同时**端到端失败无闭环**，E2E 红了之后人工分类修代码效率低。

v1.8 + v1.9 合并要补三件事：

1. **事中深度**（v1.8）· 让 AI 在架构/开发/测试的**过程**中按 AC 分行工作（不是做完再标注）
2. **验收深度**（v1.8）· 对"所开发的业务场景"做 100% 机械可验的形式覆盖（= `verification_matrix` 行数 == `@CoversAC` 测试方法数）
3. **失败闭环**（v1.9）· E2E 红了自动分诊 · 对应 Agent 自动修 · 超限升级人审

**附带并入**：**视觉回归**（方向 3）作为 `verification_matrix` 的一类验证点 · 不单独立项。

**不纳入本期的**：生产告警回流 Synthetic E2E（v2.0 起）· Matrix Challenger 对抗性验证（v2.0 候选 · v1.9 预留接口）。

### §27 新节 · E2E 失败闭环（三 Agent 协同）

**ASCII 协同图 + 六类分诊表 + Fix Agent 权限/循环/回跑要求 + needs-human 四种 User 处置 + Matrix Challenger 预留位 + reports/triage/<failure_id>.yml schema** · 详见主文 §27。

**六类分诊**：`code_bug_backend` / `code_bug_frontend` / `test_flake` / `matrix_ambiguity` / `infra_issue` / `visual_drift`。

**confidence 规则**：
- `confidence ≥ 0.8` · 自动派 Fix Agent
- `0.6 ≤ confidence < 0.8` · User 确认后派
- `confidence < 0.6` · 直接升级 `needs-human`

**Fix Agent 硬约束**：
- 写权限白名单 = Triage 报告的 `related_files` + 对应测试文件
- commit message `[fix-e2e][<failure_id>] <summary>`
- `fix_attempts > 3` 强制停手 + 升级 `needs-human`
- 修完必跑 E2E 该用例回归 · 回归绿才算 `resolved`

**needs-human 四种 User 处置**：(a) 接受当前结论（`needs-human-accepted`）· (b) 裁决 matrix 歧义 → 回 §26.1 Step 2 · (c) 降级为 `wontfix`（需附豁免理由 + 登记 §23）· (d) 打 Hotfix 分支（历史遗留 bug · 不算 E2E 失败闭环失败）。

## 六、§26 · Retrofit 流程（v1.8 · 插在 §25 之后 · §27 之前）

**适用范围**：凡已打 `<phase>-done` tag 的 Phase（按 v1.5/v1.6/v1.7 旧规则完成）。未打 `-done` 的 Phase 不走 Retrofit · 直接按 v1.8 新规则继续。

**六步流程**：
1. `retrofit-to-v18.sh <phase>` · 打 `<phase>-retrofit-start` tag · Analyst 语义扫描产 draft
2. User 审 `reports/retrofit/<phase>-plan.md`（arch diff · matrix 行数 · critical AC 清单 · 预期复写清单 · 补测数量）
3. Builder Agent 按 plan 打 patch（加注解 / 新测试 / arch 重组 · 统一 `[retrofit/<phase>][SC-XX-ACY]` commit 前缀）
4. Verifier Agent **100% 独立复写**（不抽样 · 这是 retrofit 代价）· 暴露旧 bug → 走 Hotfix 独立分支 · 不算 Retrofit 失败
5. 跑 V-SX-20 含 retrofit 豁免（`--commits` 只校验 retrofit-start tag 之后的增量）
6. 全绿打 `<phase>-v1.8-compliant` tag（与 `<phase>-done` 共存）

**四条豁免**：
1. commit history 豁免 · 不回写前缀 · 防破坏 blame
2. `--commits` 仅校验 retrofit 期间增量 · 传 `--since <retrofit-start-commit>`
3. arch.md 原地重写不 bump ADR · 视为追溯合规而非架构迭代
4. Verifier 暴露旧 bug 走 Hotfix 不算 Retrofit 失败

**Batch A / Batch B 并行策略**：
- **Batch A**：v1.8 升级后新开工 Phase（S7 / S8 / S11）· 直接按 v1.8 做
- **Batch B**：v1.8 升级前已 `s{n}-done` 的 Phase（S3 / S4 / S5）· 走 §26 Retrofit
- 并行约束：两 Batch 互不阻塞 · S9 Release Gate 要求 Batch A + Batch B 全绿才能打 `release-v1.0.0`

**前置关系**：Retrofit 必须在 v1.9 的 S9 E2E Runner 扩容前完成 · 否则 Runner 的 matrix 策略没有合规的 verification_matrix 可读。
