# 模块关系分析 — 日历模块 × 复习模块

> 版本：v1.1 · 2026-04-21（对齐方案 β · 今日聚合派）
> 适用：Longfeng AI 错题本项目
> 目的：作为 AI / 架构师 / 新入职开发的权威参考，明确"日历"与"复习"两个模块的边界、依赖、数据流与 UX 关系
> 基线文档：`业务与技术解决方案_AI错题本_基于日历系统.md` v1.2（§2A / §3 / §7 / §8 / §9）

## 版本差异（v1.0 → v1.1）

| 维度 | v1.0 日历派 | v1.1 今日聚合派（方案 β） |
|---|---|---|
| 第一 Tab | 日历（月视图） | **首页 P-HOME**（今日聚合） |
| 日历呈现 | 顶级 Tab 页 | 降级为**二级页 P10** `/calendar/month` |
| 心智模型 | 时间视角优先 | **任务队列视角优先** + 时间视角补强 |
| 视图融合锚点 | 无（日历与复习各自独立） | **事件详情 P11 双形态同壳**（复习节点 / 通用事件） |
| 入口冗余 | 日历 Tab 单一入口 | **三个入口**：首页周条带 / 首页快捷入口 / 我的 Tab |
| 后端契约 | 不变 | 不变（复习 → 日历 Feign 仍为唯一写入路径） |

> **要点**：后端契约与数据模型**完全不变**，只是前端把"日历心智"从 Tab 降级到二级页，同时用一个"双形态事件详情"让学生理解"一个节点同时是一条学习任务也是一个日历事项"。

## TL;DR（一句话版本）

> **复习模块（review-plan-service）是业务大脑，负责"什么时候该复习什么"；日历模块（calendar-core / reminder / notification）是展示与触达平台，负责"把这件事按时、按端、按偏好告诉学生"。**

两者是「**基础设施 × 领域业务**」的分层关系，而不是并列功能。复习模块单向依赖日历模块，日历模块对"错题复习"无任何特殊认知。

---

## 一、职责边界（各管什么）

### 1.1 复习模块 · 业务层（review-plan-service）

专注**艾宾浩斯遗忘曲线**的业务语义：

- 根据《艾宾浩斯.md》7 节点模型（T0 初记 / T1 1h / T2 1d / T3 3d / T4 7d / T5 15d / T6 30d）为每张错题生成一份 `ReviewPlan` + 一串 `ReviewNode`
- 学生每次自评后根据 `MASTERED / PARTIAL / FORGOT` 三种信号推进或重置节点
- 计算题目掌握度、管理题目归档、推荐变式、输出能力雷达
- 是整个学习业务的 **Source of Truth**

### 1.2 日历模块 · 平台层（calendar-core / reminder-service / notification-service）

专注**时间与展示**的通用能力：

- 管理 `calendar_event`、`calendar_reminder` 两类通用表
- 月 / 周 / 日视图、时区处理、跨端同步、订阅共享
- 提醒扫描、多渠道推送（小程序订阅消息 / APP Push / 邮件 / 短信）
- **不关心** 挂上去的是"错题复习"、"家庭生日"还是"吃药提醒"——只认 `event.startAt / endAt / relationType / relationId`

## 二、集成方式（怎么对接）— 三个关键选择

### 2.1 单向依赖，不侵入日历表

复习模块依赖日历模块，反之**不成立**。

- `wb_review_node` 是复习侧的事实表
- `calendar_event` 是日历侧的投影
- **错题域永不直接写** `calendar_event`，只能通过日历的 internal Feign API（`POST /internal/events/batch`）

**收益**：未来想换 CalDAV、苹果日历或 Google Calendar 对接时，复习业务零感知。

### 2.2 通过"多关联机制"挂接，不新建外键

日历事件通过以下两个字段认领复习节点：

```
relation_type = "STUDY"
relation_id   = "question:{qid}:node:{nid}"
```

日历表结构**完全不用改**，等于把"错题复习"当成日历的一种子类型，**天然支持未来接入其他学习类事项**（背单词、刷网课、读论文等）。

### 2.3 事件驱动双向联动

| 触发点 | 方向 | 通道 |
|---|---|---|
| 错题保存 → 需要排期 | review-plan → calendar-core | Feign `POST /internal/events/batch` |
| 节点到期 → 需要提醒 | calendar-reminder → notification | RocketMQ `notification.push.topic` |
| 日历点事项 → 打开复习 | 前端 URL Scheme `longfeng://review?nodeId=` | 深链透传 |
| 学生完成自评 → 变更事项颜色/状态 | review-plan → calendar-core | Feign `PATCH /internal/events/{id}/state` |
| 错题归档 → 级联取消未来节点 | wrongbook → review-plan → calendar-core | 级联事件 `question.archived.topic` |

## 三、典型数据流 — "一张错题的一生"

```
  学生保存错题 P03
       ↓
  wrongbook-svc 发 question.created 事件
       ↓
  review-plan-svc 生成 ReviewPlan + 7 个 ReviewNode (T0..T6)
       ↓  Feign 批量
  calendar-core 落 7 条 calendar_event (relation_type=STUDY)
       ↓  时间到
  calendar-reminder(XXL-Job) 每分钟扫 ready_at
       ↓  RocketMQ
  notification-service 按用户偏好 → 小程序订阅消息 / APP / 邮件
       ↓  学生点击推送
  深链 longfeng://review?nodeId=... → P07 复习执行
       ↓  学生自评 FORGOT
  review-plan-svc 取消后续 T2..T6 + Feign 让 calendar-core 删除对应 event
       ↓  重排从 now 开始的新 T0
  calendar-core 重新落 7 条新 event
```

整条链路里，**日历是"被告知"的那一方**，复习才是发号施令的那一方。

## 四、前端如何体现这种关系（方案 β · 今日聚合派）

前端采用 **5 Tab 结构** + **1 个视图融合锚点**：

```
Tab Bar：[首页 P-HOME] [错题本 P05] [拍题⊕ P02] [复习 P07] [我的 P13]
                                                          
二级融合层：                P10 日历月视图  ──►  P11 事件详情（双形态同壳）
                                                           ├─ 复习节点形态
                                                           └─ 通用事件形态
```

### 4.1 三个核心心智入口

| 入口 | 页面 | 心智模型 | 主要操作 |
|---|---|---|---|
| **Tab 1 · 首页** | P-HOME | "**今日任务中心**"（task-centric） | 今日复习大卡"全部开始" → P07/P08 / 七日条带扫全周 / 消息聚合 |
| **Tab 4 · 复习** | P07 | "**队列视角**" | 分时段排序 · 单题/批量开始 · 直达 P08 |
| **二级页 · 日历** | P10 | "**时间视角**" | 月/周/日切换 · 浏览 · 宏观规划 · 显示复习开关 |

> **关键决策**：日历由 Tab 降级为二级页（P10），由 P-HOME 承担"每天第一屏"的心智角色。日历仍是学生主动"调取"时间视角的重要工具，但不是每日必经路径。

### 4.2 事件详情 P11 · 视图融合锚点（v1.1 新增）

P11 是"日历 × 复习"关系的 **UX 层唯一表达**。它根据 `relation_type` 渲染**双形态**，但共享同一套壳（header / 时间&提醒卡 / 关联对象卡 / action bar / tab bar）：

| 形态 | 触发条件 | 独有模块 | 共享壳 |
|---|---|---|---|
| **复习节点形态** | `relation_type = STUDY` | T 级胶囊 · 错题缩略 · 6 节点记忆曲线 · 留存率预测 · 下次排期 · "立即复习"CTA → P08 | 顶部导航 · 时间&提醒 · 关联对象 · tab bar |
| **通用事件形态** | `relation_type ∈ {EXAM, FAMILY, REMIND, …}` 或空 | 事件类型 icon · 地点 · 参与人 · 自定义重复规则 · "编辑/删除" | 顶部导航 · 时间&提醒 · 关联对象 · tab bar |

**约定**：同一个文件 `11_event_detail.html` 承担两种形态；当 `relation_type = STUDY` 时自动展开"记忆曲线 / 错题缩略"区域，否则折叠，为通用字段让位。这种"双形态同壳"是让学生理解"一个节点既是学习任务也是日历事项"的关键设计。

### 4.3 同一节点的三个可达路径

学生可从三个入口抵达同一复习节点（后端是同一条 `wb_review_node` 记录）：

1. **队列路径**（最快）：首页"今日复习大卡"全部开始 → P08 / P07 单题 → P08
2. **日历路径**（时间视角）：首页本周条带某日 → P10 月视图 → 某日事件列表 → P11 复习节点形态 → "立即复习" → P08
3. **推送路径**（被动触达）：通知 → P12 通知中心或直接深链 → P11 复习节点形态 → P08

三条路径最终都落到 P08，底层是同一个 `POST /api/review/nodes/{nid}/open` 调用。

## 五、为什么要这样切（设计收益）

| 收益 | 具体体现 |
|---|---|
| **不重复造轮子** | 时区、跨端同步、订阅共享、免打扰时段、多渠道推送全部直接复用 |
| **业务解耦** | 未来加"背单词计划"、"网课提醒"只需再写一个领域模块，同样挂 `STUDY` 类型，不动日历 |
| **可替换性** | 日历引擎换 CalDAV 或接入 Apple/Google Calendar 时，复习业务零改动 |
| **单一数据真源** | 节点的业务语义只在 review-plan 里；展示层从不存储业务状态 |
| **事件驱动** | 通过 RocketMQ 解耦，推送失败 / 重试 / 补偿都在平台层统一兜底 |

## 六、必须警惕的边界问题

### 6.1 "完成态"的语义差异

日历事件与复习节点的状态是**两套独立但有映射关系**的状态机：

| 复习节点 grade | 日历事件 state | 展示颜色 |
|---|---|---|
| `MASTERED`（已掌握） | `COMPLETED` | 绿色 |
| `PARTIAL`（部分） | `COMPLETED` | 橙色 |
| `FORGOT`（遗忘） | `COMPLETED`（当前节点）+ 级联取消后续 + 新建 7 条新事件 | 橙红色 |

**约定**：日历永远只知道"这件事做完了没"，具体"做得怎样"由复习模块决定并回写颜色。

**禁止**：任何开发不得绕过 review-plan-service 直接修改 `calendar_event.state` 或颜色，否则业务规则会失效。

### 6.2 不要在日历表上加错题专属字段

如果有人想在 `calendar_event` 加 `mastery_pct`、`kp_list`、`difficulty` 等字段——**一定拒绝**。这是让日历"学习化"的开始，会破坏平台层的通用性。正确做法是通过 `relation_id` 反查 `wb_question` / `wb_review_node`。

### 6.3 不要让日历推送越级决定复习业务

推送失败或被拒绝不应该影响复习节点状态。日历层只负责"尝试告知"；"是否真的完成复习"由学生在 P07 里的自评决定。两件事的记录互相独立（`push_log` vs `review_record`）。

## 七、未来演进空间

保持当前分层关系，未来可以低成本扩展：

1. **接入系统日历**：在 `calendar-core` 外挂一层 CalDAV adapter，学生可以在 Apple / Google Calendar 里看到复习事项，**零改动** review-plan。
2. **多业务叠加**：背单词、刷题计划、网课打卡都可以挂 `relation_type=STUDY`，共享同一套推送 / 同步 / 多端机制。
3. **家长 / 老师视图**：开日历的只读订阅权限，家长看到学生的复习日程，**无需**触碰学习业务数据。
4. **海外版合规**：日历数据本地化存储、复习数据集中分析——两者物理分离可分别遵守不同合规要求。

## 九、v1.1 变更清单（给开发的行动项）
