### 技术契约（Style Dictionary 4.x）

#### 1) 依赖

- 必用 `style-dictionary@^4.0.0`（v4 的 ESM、platforms、formats、transforms 皆已稳定）
- 禁止引入除 `style-dictionary` 以外的运行期依赖；如需 HSL 调整可用 Node `node:` 内置或手写 6 行 hex→hsl→hex 函数

#### 2) `build-tokens.mjs` 必须包含 4 个平台产物

```js
// 伪代码结构（不要照抄值，需根据实际 JSON 内容推导）
import StyleDictionary from 'style-dictionary';

// 注册自定义 transform: 仅对 color 类 token，按 platform_overrides 调整饱和度
StyleDictionary.registerTransform({
  name: 'color/wechat-saturation-boost',
  type: 'value',
  transitive: true,
  filter: token => token.$type === 'color' || token.type === 'color',
  transform: (token) => {
    // 读 color.json._meta.platform_overrides.wechat.saturation_delta
    // 对 token.value 做 HSL.s += delta，hex 回填；clamp 到 [0,100]
    // 保留原 alpha
    return adjustSaturationHex(token.value, SAT_DELTA_PCT);
  }
});

// 注册自定义 format for TypeScript（扁平化为嵌套对象 + as const）
StyleDictionary.registerFormat({
  name: 'ts/nested-const',
  format: ({ dictionary }) => {
    const tree = dictionary.tokens; // SD4 直接给 nested
    return `/* AUTO-GENERATED by build-tokens.mjs. Do not edit. */
export const tkn = ${stringifyAsConst(tree)} as const;
export type Tkn = typeof tkn;
`;
  }
});

export default {
  source: ['../../../design-system/tokens/*.json'],
  platforms: {
    css: {
      transformGroup: 'css',
      buildPath: 'src/',
      files: [{
        destination: 'tokens.css',
        format: 'css/variables',
        options: { outputReferences: true, selector: ':root' }
      }]
    },
    wxss: {
      transforms: ['attribute/cti','name/kebab','color/wechat-saturation-boost','color/hex'],
      buildPath: 'src/',
      files: [{
        destination: 'tokens.wxss',
        format: 'css/variables',
        options: { outputReferences: false, selector: 'page' }
      }]
    },
    ts: {
      transformGroup: 'js',
      buildPath: 'src/',
      files: [{ destination: 'tokens.ts', format: 'ts/nested-const' }]
    }
  }
};
```

> 结构示意而已——你必须根据 6 份 JSON 的实际形状、命名前缀调整。**禁止** 修改 JSON 里的 value，只能做 transform。

#### 3) 命名契约

- CSS/WXSS: `--tkn-<group>-<name>-<variant>`（从 JSON 的路径派生，`color.blue.500` → `--tkn-color-blue-500`）
- TS: 嵌套对象路径一致（`tkn.color.blue[500]`）
- 不能出现 `--color-*`、`--tkn_*`、camelCase 等偏差

#### 4) 小程序饱和度 +12% 的唯一事实源

- 只读 `design-system/tokens/color.json` 的 `_meta.platform_overrides.wechat.saturation_delta`
- 如该字段不存在或非数字 → **报错并退出**，不得默认 0
- 仅作用于 wxss 平台，tokens.css / tokens.ts **不得** 被污染
- 非 color 类 token 不受影响

#### 5) 构建脚本

在 `frontend/packages/ui-kit/package.json` 加：
```json
{
  "scripts": { "build:tokens": "node build-tokens.mjs" },
  "devDependencies": { "style-dictionary": "^4.0.0" }
}
```

## 给父 Agent 的元信息（不传给 Builder，但保留在本文件供回溯）

- **为什么 Step 3 独立成 task**: Style Dictionary 4.x 的 transform/format 定制是一次性基础设施，混在 Step 2 会让 token 决策被工具耦合。独立后 token 语义与编译工具解耦，将来换 Tokens Studio / Theo 只换 Step 3。
- **为什么要强制 sha256 校验**: Sd 阶段多次反复迭代，Step 2 一旦被手改过，Step 3 产出的 tokens.css 会与 `apple-base.decision.md` 脱节，造成前端/文档不一致的隐性事故。
- **为什么不在 wxss 走 `rpx`**: 小程序 2.10+ 已支持 `var(--xxx)`，且 rpx 是运行时视口换算，构建期不应做假；保留 px，由组件层按需 rpx 化。
- **为什么 tokens.ts 用 `as const` + 嵌套对象而不是 flat record**: Konsta UI / Vant Weapp 两侧都能消费嵌套结构，`tkn.color.blue[500]` 比 `tkn['color-blue-500']` 更利于 IDE 自动补全；`as const` 让 `Tkn` 类型精确到字面量，下游可以收紧 props 类型。
- **饱和度 +12% 为什么只作用于小程序**: Apple HIG 默认在大色域 P3 显示器上足够鲜艳；微信小程序主要跑在中端 Android sRGB 屏 + 微信 WebView 色彩压缩，叠加 +12% 可抹平体感差距。此决策落在 Step 2 的 `_meta.platform_overrides`，Step 3 只是执行。
